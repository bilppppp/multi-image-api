====================
第二部分：实现 Tuzi /images/edits
====================

在 Tuzi 图生图路径上，实现或修复：

POST https://api.tu-zi.com/v1/images/edits

要求：
1. 使用 multipart/form-data
2. 如果 provider 里 model 为 openai/gpt-image-2，实际发给 Tuzi 时使用 gpt-image-2
3. 至少支持 1 张主图，最好兼容多图
4. image 字段使用重复追加的 image，不要改成 image[]
5. 如支持 mask，则字段名为 mask

请求字段要求：

1. model
- 最终发给 Tuzi 时为 gpt-image-2

2. prompt
- 使用用户的图生图 / 图片编辑提示词

3. image
- 支持 1 张或多张输入图
- 第一张视为主编辑图
- 第一张主图需要规范化为方图 PNG
- 如果有多张，后续图片作为参考图

4. mask
- 对非方图输入，必须生成与主图方图同尺寸的 mask
- mask 字段名固定为 mask
- mask 与 image 尺寸一致
- mask 用于区分原图有效内容区域和补边区域
- 如果 OpenClaw 当前已有单独 mask 输入能力，也应兼容用户提供的 mask

5. size
注意：Tuzi edits 的 image / mask 可以是方图，但 size 不应固定为方图。
size 应代表用户想要的最终输出尺寸。
不要再把 Tuzi 图生图请求里的 size 固定为 squareSide x squareSide。
应新增或修正 outputSize / targetSize 决策逻辑。
form.set("size", ...) 应使用 outputSize，而不是 squareSide x squareSide。

outputSize 优先级：

A. 用户显式指定 size 时，直接使用，例如：
- 1536x1024
- 1024x1536
- 2048x1152
- 1024x1024

B. 用户显式指定比例或方向时：
- 16:9 / 横版 / 横幅 / 横图 / landscape -> 1536x1024
- 9:16 / 竖版 / 竖幅 / 竖图 / portrait -> 1024x1536
- 1:1 / 方图 / 方形 / square -> 1024x1024
- 4:3 -> 1536x1152
- 3:2 -> 1536x1024

C. 用户说“保持原图比例 / 参考原图比例 / 按原图比例”时：
- 根据第一张原图宽高比生成 outputSize
- 宽高必须为 16 的整数倍
- 最大边 <= 3840
- 长短边比例 <= 3:1
- 尽量接近原图比例

D. 用户什么比例和尺寸都没说时：
- 非方图输入：默认保持原图比例生成 outputSize
- 方图输入：默认 outputSize = 1024x1024

举例：
- 输入图 1536x1024，用户没指定比例 -> size=1536x1024
- 输入图 1536x1024，用户说“保持原图比例” -> size=1536x1024
- 输入图 1536x1024，用户说“9:16 竖幅” -> size=1024x1536
- 输入图 1536x1024，用户说“16:9 横版” -> size=1536x1024
- 输入图 4:3，用户没指定比例 -> size=1536x1152 或其他符合 4:3 且宽高为 16 整数倍的尺寸

6. quality
- 用户显式指定 quality，则透传用户值
- 用户未显式指定，则 Tuzi 图生图默认使用 "low"
- 不要默认使用 "auto"

7. n
- 默认设置为 1

8. response_format
- 固定设置为 "url"

9. timeout
- Tuzi 图生图请求 timeout = 1800000 ms（30 分钟）

====================
第三部分：第一张主图输入规范化
====================

仅在以下条件同时满足时启用：
- baseUrl 包含 api.tu-zi.com
- 当前请求是 /images/edits
- 当前至少有 1 张输入图片

对第一张主图做规范化：

1. 读取原图宽高：
- origWidth
- origHeight

2. 计算 squareSide：
- 供 image / mask 的方图尺寸使用
- 优先取原图长边
- 最大不超过 3840
- 必须是 16 的整数倍
- 不要无条件固定为 1024
- 不要无条件固定为 1536

3. 同时满足 Tuzi edits 输入限制：
- image 为 PNG、方形、小于 4MB
- mask 为 PNG、方形、小于 4MB

4. 若 image 或 mask 任意一个 >= 4MB：
- 不要直接失败
- 自动按 16 的整数倍逐级回退 squareSide
- 直到 image 和 mask 都 <4MB
- 尽量保持较高分辨率
- 可按顺序尝试：
  3840、3584、3328、3072、2816、2560、2304、2048、1792、1536、1280、1152、1024
- 一旦较大尺寸已满足 <4MB，就不要继续缩小

5. 规范化第一张主图：
- 转为 PNG
- 转为 RGBA / 带 alpha 通道
- 等比缩放，完整放入 squareSide x squareSide 方图画布
- 居中放置
- 不允许直接拉伸成 1:1
- 处理成方形
- 上传时字段名仍为 image

6. 对非方图输入，必须记录 contentRect：
- origWidth
- origHeight
- contentX
- contentY
- contentWidth
- contentHeight
- squareSide

7. 多图规则：
- 第一张作为主图，进行规范化
- 后续参考图尽量保持原样
- 字段名继续重复使用 image

如果当前环境已有 sharp / jimp / canvas / ImageMagick 等可用图像处理能力，优先使用已有依赖；
不要安装新依赖；
如果当前环境没有可用图像处理能力，不要写无法运行的代码，请如实报告。

====================
第四部分：mask 与透明区域策略
====================

图生图路径上，请采用更稳的策略，避免黑色 mask / 黑底 / 白边 / 方图留边：

1. 对非方图输入：
- image 为补成方图后的主图
- mask 为同尺寸 PNG
- image 和 mask 均使用 squareSide x squareSide
- mask 字段名固定为 mask

2. mask 规则：
- 原图有效内容区域 contentRect：标记为允许编辑区域
- 补边区域：标记为不允许编辑区域
- mask 必须与 image 尺寸一致

3. 若当前实现采用 alpha mask：
- 需要编辑的区域使用透明区域
- 不需要编辑的区域使用不透明区域

4. 若当前实现采用黑白 mask：
- 必须与 Tuzi 实测可用格式一致
- 不要随意反转黑白含义
- 如需改动，只能在 Tuzi edits 路径上做最小修改

5. 若已有单独 mask 能力：
- image 使用正常主图
- mask 单独上传
- 不要过度依赖“主图大面积透明区域”作为唯一编辑方式

6. 若当前没有现成 mask 输入能力：
- 必须至少支持自动生成 mask
- 不要为 mask 大改架构
- 保持 Tuzi edits 路径可用

====================
第五部分：图生图响应处理
====================

Tuzi /images/edits 成功返回常见格式：
{
  "data":[{"url":"https://..."}],
  "created":...,
  "usage":{...}
}

请确保：

1. 如果响应中存在 data[i].url：
- 读取 url
- 下载图片
- 将下载得到的图片按 OpenClaw 现有 image result 结构返回

2. 保持兼容 b64_json：
- 如果存在 b64_json，保留兼容逻辑
- 如果同时存在 b64_json 和 url，可优先保持原有逻辑，但不要破坏 url 下载逻辑

3. 保留已跑通的 Tuzi 文生图 response_format="url" 和 data[].url 下载逻辑

4. 下载 Tuzi 返回结果后，必须读取返回图实际尺寸

5. 如果返回图实际尺寸已经等于 outputSize：
- 不要再按 contentRect 裁切
- 不要再强行 crop 成原图区
- 直接进入最终整理流程，例如必要的白底铺平

6. 只有以下情况才允许使用 contentRect 进行旧式补救裁切：
- 返回图仍然是 squareSide x squareSide
- 且本次目标 outputSize 不是方图
- 且确实需要从方图结果中补救式裁回

7. 不要无条件执行 contentRect 裁切

8. 如果执行 contentRect 裁切：
- 必须在白底铺平之前执行
- 不要使用会导致明显拉伸变形的强制 resize
- 如必须 resize，优先保持比例

====================
第六部分：自动铺白底，解决黑底 / 黑 mask
====================

Tuzi 图生图成功后，下载到的图片有时带 alpha 通道，预览时会出现黑底 / 黑 mask / 黑框。
因此请在 Tuzi /images/edits 路径上增加：

1. 若最终输出图是带 alpha 通道的 PNG：
- 在返回给用户前，自动将透明区域铺到白色背景上
- 生成一个不透明的最终图片文件
- 优先保持 PNG 输出；如当前链路更适合，也可输出不带透明通道的常规图片

2. 目标：
- 用户最终看到的图不再出现黑底 / 黑 mask / 黑框
- 不改变主体内容
- 仅修复透明显示问题

3. 执行顺序：
- 先完成必要的尺寸判断 / 必要裁切
- 再执行白底铺平
- 不要先铺白底再无条件返回 1:1 方图

4. 若当前环境已有 sharp / jimp / canvas / ImageMagick 等可用图像处理能力，优先使用已有依赖；
不要安装新依赖
