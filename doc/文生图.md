第二部分：Tuzi 文生图 provider 补丁

- model 为 gpt-image-2 / openai/gpt-image-2，或 model 未显式传入但 baseUrl 是 api.tu-zi.com

--------------------------------
一、超时
--------------------------------
1. Tuzi 图片生成请求 timeout = 1800000 ms，即 30 分钟
2. 只影响 Tuzi 图片生成路径，不影响聊天、Codex、文本模型或其他 provider

4. data[].url 下载阶段也使用 1800000 ms，仅限 Tuzi 图片结果下载

--------------------------------
二、response_format=url
--------------------------------
1. 调用 Tuzi /v1/images/generations 时，请求 body 自动加入：
   response_format: "url"
2. 不影响非 Tuzi provider、普通聊天和文本请求

--------------------------------

--------------------------------
1. 如果响应中存在 data[i].url：

   - 将图片 buffer/bytes 按 OpenClaw 现有 image result 结构返回
2. 继续兼容 b64_json
3. 如果同时存在 b64_json 和 url，优先保持原有 b64_json 逻辑
4. 图片 URL 下载失败时保留关键错误信息

--------------------------------
四、尺寸、比例与 aspectRatio
--------------------------------

背景：
原始逻辑可能类似：
const size = req.size ?? DEFAULT_SIZE;
DEFAULT_SIZE = "1024x1024";
这会导致未显式传 size 时默认 1:1。
另外，当前 OpenAI-compatible 图片 provider 不支持直接透传 aspectRatio，否则可能报：
openai generate does not support aspectRatio overrides.

总原则：
1. 用户提示词里的比例只用于本地推导最终 size，不要向 Tuzi 透传 aspectRatio
2. 不要因为用户写 2:1、16:9、9:16、4:3、3:2 等比例，就回复“比例不支持”“改用 21:9”“只能用 1536x1024”
3. 应直接在内部换算成合法 size 后继续生成
4. 如果同时有显式 size 和比例/方向/aspectRatio，以显式 size 为准
5. 发给 Tuzi 的请求应使用 size，不使用 aspectRatio
6. 这套逻辑只针对 Tuzi 文生图 /images/generations 路径

A. 显式 size 优先
如果 image_generate 参数已有 size，或 prompt 中能识别显式尺寸，例如：
1024x1024、1024*1024、1024×1024、1536x1024、2048x1024、1536x1152、864x1536 等，
则原样透传该 size。
显式尺寸提取至少支持 x、*、×。
如果 size 和 aspectRatio 同时存在：保留 size，删除 aspectRatio，不抛错。

B. 常见比例映射
如果没有显式尺寸，但 prompt 或 aspectRatio 中包含比例/方向，则映射为具体 size：

- 1:1 -> 1024x1024
- 3:2 -> 1536x1024
- 2:3 -> 1024x1536
- 4:3 -> 1536x1152
- 3:4 -> 1152x1536
- 16:9 -> 1536x864
- 9:16 -> 864x1536
- 2:1 -> 2048x1024
- 1:2 -> 1024x2048
- 21:9 -> 1344x576
- 9:21 -> 576x1344
- 3:1 -> 3072x1024
- 1:3 -> 1024x3072

方向词默认映射，仅在用户没给明确比例时使用：
- 横版 / 横幅 / 横图 / landscape -> 1536x864
- 竖版 / 竖幅 / 竖图 / portrait -> 864x1536
- 方图 / 方形 / square -> 1024x1024

如果比例词和方向词同时出现，以比例词为准，不要让“横版/竖版”覆盖 2:1、4:3、3:2 等明确比例。

C. 任意 a:b 比例自动换算
如果比例不在上表，如 5:4、7:5、11:8、13:10、8:5、5:8 等：
1. 解析 a:b，ratio = a / b
2. ratio > 1：默认短边 1024，长边按 ratio 计算，并四舍五入到最接近的 16 整数倍，size="{longEdge}x1024"
3. ratio < 1：默认短边 1024，长边按 1/ratio 计算，并四舍五入到最接近的 16 整数倍，size="1024x{longEdge}"
4. ratio = 1：size="1024x1024"
5. 结果必须满足：宽高为 16 的整数倍、最大边 <= 3840、长短边比例 <= 3:1、总像素数满足 Tuzi 文档要求、尽量接近原比例
6. 如果短边 1024 推导不合法，可在 1024、1152、1280、1536 等候选短边中择优
7. 不要生成解释性回复，不要说“我改用了最接近的 21:9”，应内部换算后直接继续生成

D. 只有方向词时
没有显式尺寸和明确比例，只有方向词：
- 横版 / 横幅 / 横图 / landscape -> 1536x864
- 竖版 / 竖幅 / 竖图 / portrait -> 864x1536
- 方图 / 方形 / square -> 1024x1024

E. 什么都没写
没有尺寸、比例、方向信息时：
size = "1024x1024"

F. aspectRatio 处理
1. 不要向下游透传 aspectRatio
2. 如果有 size：保留 size，删除 aspectRatio
3. 如果无 size 但有 aspectRatio：将 aspectRatio 当作比例输入，按 B / C 转为 size，然后删除 aspectRatio
4. 如果 prompt 里也有比例词，如“2:1 横幅”“16:9 横版”“9:16 竖版”“4:3 海报”，且工具层又传 aspectRatio，也统一转成 size，删除 aspectRatio
5. 不要因为 aspectRatio 存在而抛错

结果要求：
1. 2:1、4:3、3:2、16:9、9:16 等比例必须优先转换为正确 size，不要统一落到 1536x1024
2. 不要把 2:1 改成 21:9
3. 不要把 16:9 改成 1536x1024
4. 不要因为 aspectRatio 报错
5. 不要扩大到非必要模块
